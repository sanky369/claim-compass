# Agent Spec

## Overview

ClaimCompass is a Google Cloud Rapid Agent Hackathon agent for the MongoDB track.

It is a revenue-recovery agent for independent healthcare providers. The golden-path demo resolves one synthetic denied therapy claim: BCBS-TX, CPT `90837`, telehealth modifier missing, denial codes `CO-45` + `N179`.

The agent reads the denial, extracts structured fields with Document AI, retrieves payer-specific playbook rules from MongoDB Atlas Vector Search through the official MongoDB MCP Server, decides the next-best action, drafts corrected-claim guidance, and writes the updated claim state plus trace events back to MongoDB.

Judge-facing wording: built with Gemini and Google Cloud Agent Builder using ADK, designed for Agent Runtime / Vertex AI Agent Engine, hosted for the demo on Cloud Run, and integrated with MongoDB MCP. Only say "deployed on Agent Runtime" if the final ClaimCompass-specific ADK app is actually deployed and tested there.

## Example Use Cases

### Golden Path

Input:

```json
{ "denial_id": "demo_denial_001" }
```

Expected behavior:

1. RootAgent reads the denial from MongoDB through MCP `find`.
2. If `ocr_status != "done"`, RootAgent calls the Document AI Form Parser tool on the uploaded EOB `gcs_uri`.
3. RootAgent updates the denial through MCP `update-many` with a unique `denial_id` filter and extracted fields.
4. RootAgent delegates to PolicyAgent with `{ "payer_id": "bcbs_tx", "cpt": "90837", "raw_text": "..." }`.
5. PolicyAgent generates a Google embedding with `gemini-embedding-2`.
6. PolicyAgent calls MongoDB MCP `aggregate` with a `$vectorSearch` first stage against `payer_playbooks`.
7. PolicyAgent calls MongoDB MCP `find` against `carc` and `rarc`.
8. RootAgent classifies the denial as `corrected_claim` with confidence.
9. RootAgent delegates to DrafterAgent.
10. DrafterAgent generates corrected-claim guidance with citation chips tied to returned `payer_playbooks._id` values.
11. DrafterAgent inserts the generated artifact through MCP `insert-many`.
12. RootAgent updates the denial through MCP `update-many` with a unique `denial_id` filter and final status/result.
13. RootAgent inserts step-level `trace_events` through MCP `insert-many`.

Expected output:

```json
{
  "denial_id": "demo_denial_001",
  "bucket": "corrected_claim",
  "bucket_confidence": 0.9,
  "plain_english": "The denial is likely caused by a missing telehealth modifier on CPT 90837.",
  "recommended_action": "Submit a corrected claim with the required telehealth modifier and supporting payer-specific guidance.",
  "generated_artifact_id": "<artifact_id>",
  "status": "triaged"
}
```

## Architecture and Sub-Agents

### RootAgent

Model: Gemini Pro class model, configured by environment variable.

Responsibilities:

- Own the denial-resolution workflow.
- Read and update `denials`.
- Call Document AI Form Parser tool.
- Delegate retrieval to PolicyAgent.
- Classify the denial into one of seven canonical buckets.
- Delegate drafting to DrafterAgent.
- Write `trace_events` throughout the run.

Tools:

- MongoDB MCP `find`
- MongoDB MCP `update-many`
- MongoDB MCP `insert-many`
- Document AI tool function: `documentai.process(form_parser, gcs_uri)`
- Gemini generation

### PolicyAgent

Model: Gemini Flash class model, configured by environment variable.

Responsibilities:

- Embed the payer/cpt/raw denial query.
- Retrieve payer playbook chunks from MongoDB Atlas Vector Search.
- Look up CARC/RARC descriptions.
- Return chunks with IDs/titles for downstream citations.

Tools:

- Gemini Embedding: `gemini-embedding-2`, `output_dimensionality=1536`
- MongoDB MCP `aggregate` against `payer_playbooks`
- MongoDB MCP `find` against `carc` and `rarc`

Vector retrieval pipeline:

```js
[
  { $vectorSearch: {
      index: "playbook_vec",
      path: "embedding",
      queryVector: "<embedding>",
      numCandidates: 100,
      limit: 8,
      filter: { payer_id: "<payer>", "scope.cpt_family": "<family>" }
  }},
  { $project: { _id: 1, title: 1, body: 1, source_url: 1, score: { $meta: "vectorSearchScore" } } }
]
```

### DrafterAgent

Model: Gemini Pro class model, configured by environment variable.

Responsibilities:

- Generate corrected-claim guidance or appeal/call-script artifact based on RootAgent bucket.
- Use only returned playbook chunks and payer reference data.
- Insert artifact into `generated_artifacts`.
- Return `generated_artifact_id`.

Tools:

- MongoDB MCP `find` against `payers`
- Gemini generation
- MongoDB MCP `insert-many` into `generated_artifacts`

Atlas Vector Search index:

```json
{
  "fields": [
    { "type": "vector", "path": "embedding", "numDimensions": 1536, "similarity": "cosine" },
    { "type": "filter", "path": "payer_id" },
    { "type": "filter", "path": "scope.cpt_family" }

}
```

Index creation path:

- Primary: MongoDB MCP `create-index`, then verify with `collection-indexes`.
- Fallback: Atlas UI if MCP index creation blocks, then still verify with MCP `collection-indexes`.

## Constraints and Safety Rules

- Runtime AI must be Google Cloud only: Gemini, Gemini Embedding, Document AI.
- Do not use Claude, OpenAI, Voyage AI runtime embeddings, AWS Textract, Mistral OCR, or other third-party AI runtime services.
- Do not use Agent Builder Data Stores, RAG Engine, or Agent Platform Vector Search for v1 retrieval. The MongoDB track grounding must visibly come from MongoDB Atlas Vector Search through MCP.
- Do not use real PHI, real patient names, payer logos, or copied payer EOB templates.
- All claim content shown in UI must be marked `DEMO DATA - NOT REAL PHI`.
- Generated guidance is decision support and must include a human-review disclaimer.
- Citation chips must resolve to real `payer_playbooks._id` values returned by PolicyAgent.
- Do not invent payer addresses, policy facts, or appeal requirements.
- If retrieved context is weak or missing, lower confidence and surface a fallback instead of hallucinating.
- Keep secrets out of git and screenshots.
- Do not build production auth for the hackathon; use a demo gate or unauthenticated demo route.

## Success Criteria

The golden path is complete when:

- The committed synthetic EOB PDF can be opened, uploaded to GCS, and processed.
- Document AI extraction writes structured denial fields back to MongoDB.
- PolicyAgent retrieves relevant playbook chunks through MongoDB MCP `aggregate` + `$vectorSearch`.
- RootAgent classifies the denial as `corrected_claim`.
- DrafterAgent generates corrected-claim guidance with valid citations.
- The denial doc is updated in MongoDB.
- A generated artifact is inserted in MongoDB.
- Trace events show Document AI, Gemini, and MongoDB MCP tool calls.
- The Next.js demo UI shows the trace panel, result, citations, and before/after MongoDB diff.
- Agents CLI eval passes the golden retrieval/classification case before UI polish.
